# План проекта ChatBot 1C v7

## Текущее состояние на 2026-07-21

Оценка готовности MVP: **40-45%**. Эта оценка основана на фактически закрытых
этапах и release gates, а не на объеме кода или числе созданных JSON-файлов.

| Фаза | Состояние | Что подтверждено | Что осталось |
| --- | --- | --- | --- |
| 0. Инициализация | завершена | чистый репозиторий, источники, допущения, версии и GitHub remote | проверить и зафиксировать private visibility перед релизом |
| 1. Бизнес-анализ | завершена | 55 FR, 14 NFR, 59 AC, 87 capabilities и корпус Q001-Q116 | изменения только через версионирование требований |
| 2. Архитектура | завершена для MVP, уточняется инкрементами | слои, переносимый skill contract, bounded LLM, evidence и package transactions | завершить schema/исполнение типизированных relational operators |
| 3. Испытания | базовая модель завершена | стратегия, независимый corpus и contract suites | реализовать все production gates и итоговый acceptance report |
| 4. Вертикальный срез | частично завершена | web/chat, DeepSeek, SQLite, JSON packages, fixture MCP, диагностика, первые 39 физических skills | локальный MCP proxy, универсальная композиция и живая проверка с 1С |
| 5. Полный каталог | начата | reference/document starter и несколько stock/order operations | canonical capability mapping, бизнес-пакеты, D01-D04 и 87/87 end-to-end coverage |
| 6. Стабилизация | не начата | только ранние regression suites | полный прогон 116 сценариев, control queries, latency и failure injection |
| 7. Поставка | не начата | alpha packaging и CI foundation | Windows/macOS install, отдельный skill release, checksums, инструкции и candidate tag |

Текущий starter содержит 39 физических skills, но только 7 его capability IDs
совпадают с обязательным каноническим набором. Остальные сгенерированы по
именам skills и не считаются продуктовым покрытием. До исправления mapping и
прохождения `RG-01` запрещено заявлять покрытие каталога как 39/87.

Живой MCP сейчас недоступен: обработка 1С требует локальный JSON-only proxy на
`http://127.0.0.1:6003`. Прокси является поставляемой частью MVP по ADR-0005.
Пока он не готов и к нему не подключена обработка 1С, выполняются fixture и
contract tests; все live/control-query gates остаются `blocked_external`.

## 1. Цель

Получить проверяемый MVP ассистента для типовой `1С:Управление торговлей`,
который:

- отвечает на практические вопросы по работе в УТ на основании документации;
- получает актуальные данные из 1С через MCP без изменения данных;
- использует заранее подготовленный набор навыков вместо разработки навыка во
  время каждого пользовательского запроса;
- выбирает и сочетает несколько навыков для составных вопросов;
- использует DeepSeek через API;
- позволяет переносить навыки между экземплярами одной конфигурации без
  обновления приложения;
- имеет web-клиент, диагностические журналы и воспроизводимые тесты.

## 2. Управление проектом

### Руководитель проекта

Отвечает за последовательность работ, постановку задач субагентам, приемку их
результатов, разрешение противоречий, управление рисками, версии, GitHub и
итоговую готовность продукта. Не подменяет архитектора техническими решениями.

### Бизнес-аналитик

Определяет пользователей, задачи, терминологию, функциональные требования,
границы MVP, каталог бизнес-навыков, пользовательские вопросы и критерии
приемки. Не выбирает архитектуру и технологии.

### Архитектор

На основании принятых требований проектирует слоистую архитектуру, контракты,
формат и жизненный цикл переносимых навыков, обработку диалога, интеграции,
наблюдаемость и способ расширения после MVP. Проверяет реализуемость и явно
возвращает руководителю нерешенные продуктовые вопросы.

### Разработчик

Реализует принятую архитектуру и продуктовый каталог навыков небольшими
вертикальными инкрементами. Добавляет эксплуатационную документацию и не меняет
контракты без решения архитектора и руководителя.

### Тестировщик

Независимо выводит тестовую модель из требований, готовит тест-кейсы и
автотесты, проверяет продукт на типовой УТ через MCP, проверяет переносимость
навыков и регистрирует воспроизводимые дефекты. Не подгоняет ожидания под
реализацию.

## 3. Порядок работ

### Фаза 0. Инициализация

Работы:

1. Создать отдельный чистый каталог и подключить новый GitHub-репозиторий.
2. Зафиксировать назначение проекта, план, допущения и правила версий.
3. Проверить доступность файловой выгрузки типовой УТ, MCP, документации и
   DeepSeek API без переноса кода прошлых версий.
4. Составить реестр входных источников с назначением и ограничениями каждого.

Результаты:

- чистый репозиторий;
- план проекта;
- реестр источников;
- журнал допущений и вопросов.

Условие перехода: источники найдены либо для отсутствующих источников определен
неблокирующий заменитель.

### Фаза 1. Бизнес-анализ

Работы:

1. Описать целевых пользователей и их основные задачи.
2. Сформировать функциональные и нефункциональные требования.
3. Определить границы MVP и явно исключенные возможности.
4. Подготовить словарь терминов типовой УТ.
5. Сформировать каталог готовых навыков по бизнес-областям.
6. Для каждого навыка определить пользовательскую ценность, входные данные,
   ожидаемый результат и примеры применения без описания реализации.
7. Подготовить не менее 80 разнотипных пользовательских вопросов, включая
   составные вопросы, продолжения диалога, отсутствие данных, неоднозначность и
   нерелевантные запросы.
8. Определить измеримые критерии приемки MVP.

Результаты:

- спецификация требований;
- словарь;
- каталог навыков;
- корпус эталонных вопросов;
- матрица требований и критериев приемки.

Условие перехода: руководитель подтвердил полноту требований, а блокирующие
вопросы разрешены.

### Фаза 2. Архитектурная проработка

Работы архитектора:

1. Спроектировать слоистую архитектуру с четкими обязанностями и контрактами.
2. Определить путь запроса от сообщения пользователя до проверенного ответа.
3. Определить разделение ответственности между приложением, DeepSeek, навыком,
   документацией, конфигурацией и MCP.
4. Спроектировать формат переносимого пакета навыка и его совместимость.
5. Определить механизмы выбора, композиции, параметризации, проверки и
   объяснения применения навыков.
6. Определить способы работы с документацией и актуальными данными 1С.
7. Определить хранение истории диалога, диагностики и воспроизводимых трасс.
8. Спроектировать расширение каталога навыков без обновления приложения.
9. Определить минимальный вертикальный срез и порядок реализации.
10. Провести саморевью архитектуры против каждого требования и известных
    неудач версии 5.

Результаты:

- архитектурное описание;
- контракты интеграций и навыков;
- решения по ключевым компромиссам;
- карта требований на компоненты;
- план реализации вертикальными срезами.

Условие перехода: по каждому обязательному требованию существует проверяемый
путь реализации, отсутствуют циклические зависимости и скрытая необходимость
писать код под каждый новый вопрос.

### Фаза 3. Проектирование испытаний

Работы тестировщика:

1. Разложить критерии приемки на позитивные, негативные и пограничные тесты.
2. Разметить эталонный корпус вопросов ожидаемым смыслом, источником,
   необходимыми навыками и правилами проверки ответа.
3. Подготовить тесты отдельных навыков, их композиции и выбора.
4. Подготовить контрактные тесты MCP и DeepSeek без зависимости от случайного
   текста модели.
5. Подготовить тест переносимости навыка между двумя чистыми экземплярами.
6. Определить метрики качества, стабильности, времени ответа и диагностируемости.
7. Разделить быстрые тесты для каждого изменения и полный приемочный прогон.

Результаты:

- стратегия тестирования;
- каталог тест-кейсов;
- машиночитаемый корпус вопросов;
- критерии блокировки релиза.

Условие перехода: тестовая модель покрывает все обязательные требования до
начала массовой реализации.

### Фаза 4. Минимальный вертикальный срез

Работы:

1. Реализовать запуск приложения и web-клиента.
2. Подключить DeepSeek и MCP в соответствии с принятыми контрактами.
3. Реализовать загрузку внешнего пакета навыков без изменения кода приложения.
4. Реализовать небольшой набор навыков, который демонстрирует:
   - ответ из документации;
   - простой запрос данных;
   - параметризованный запрос;
   - композицию нескольких навыков;
   - продолжение диалога.
5. Реализовать логи и диагностический пакет одного запроса.
6. Выполнить автоматические и живые проверки с типовой УТ.

Условие перехода: вертикальный срез проходит согласованные тесты и доказывает
переносимость одного и того же пакета навыков между экземплярами.

### Фаза 5. Полный каталог навыков MVP

Работы:

1. Реализовывать навыки группами по бизнес-областям УТ.
2. Для каждого навыка добавлять контракт, примеры, тестовые параметры,
   ожидаемую форму результата и диагностическое описание.
3. Добавить навыки поиска по документации и формирования ответа с источниками.
4. Добавить составные навыки только через композицию уже проверенных атомарных
   возможностей.
5. Проверять отсутствие дублирующих и чрезмерно узких навыков.
6. Собирать переносимый версионируемый пакет навыков независимо от приложения.

Условие перехода: обязательный каталог реализован, каждый навык имеет тесты и
не требует изменения приложения для переноса.

### Фаза 6. Интеграция и стабилизация

Работы:

1. Запустить полный корпус вопросов на живой типовой УТ.
2. Повторить каждый недетерминированный сценарий несколько раз.
3. Проверить диалоговый контекст и составные вопросы.
4. Проверить корректность запросов 1С и фактических результатов по независимым
   контрольным выборкам.
5. Проверить поведение при таймаутах, ошибках DeepSeek, MCP и отсутствии данных.
6. Проверить, что логи позволяют восстановить все решения без догадок.
7. Исправить дефекты и повторять полный прогон до прохождения порога приемки.

Условие перехода: нет блокирующих дефектов, достигнуты метрики MVP, результаты
воспроизводимы.

### Фаза 7. Поставка MVP

Работы:

1. Подготовить установку и обновление приложения.
2. Подготовить отдельную установку и обновление пакета навыков.
3. Описать настройку DeepSeek, MCP, документации и экземпляра УТ.
4. Зафиксировать ограничения прототипа и известные дефекты.
5. Собрать версионированный релиз и контрольные суммы.
6. Опубликовать исходники, тег и документацию в частном GitHub-репозитории.

Результаты:

- воспроизводимый релиз приложения;
- отдельный пакет навыков;
- инструкция установки и тестирования;
- отчет о приемке;
- точка отката.

## 4. Стратегия навыков как продукта

До архитектурного этапа фиксируются только продуктовые свойства:

- навык решает одну понятную задачу или предоставляет одну переиспользуемую
  возможность;
- значения пользователя являются параметрами, а не причиной создавать новый
  навык;
- составной вопрос использует несколько навыков;
- навык содержит достаточно описания для осмысленного выбора;
- навык можно экспортировать, проверить и импортировать без изменения кода;
- совместимость навыка с конфигурацией и версией базы проверяется до применения;
- неправильный навык можно отключить или заменить независимо от приложения;
- переносимость подтверждается автоматическим тестом на чистом экземпляре.

Точный формат и механизм исполнения определяет архитектор.

## 5. Контроль качества

Обязательные уровни проверки:

1. Проверка требований на непротиворечивость.
2. Архитектурное саморевью и ревью руководителем.
3. Автотесты разработчика для каждого инкремента.
4. Независимые тесты тестировщика.
5. Контрактные тесты DeepSeek, MCP и пакетов навыков.
6. Интеграционные тесты на живой типовой УТ.
7. Повторяемые диалоговые тесты на эталонном корпусе.
8. Ручная бизнес-проверка выборки ответов и запросов 1С.

Дефект исправляется на том уровне, где возникла общая причина. Запрещено
закрывать системный дефект частным условием под один вопрос или объект УТ.

## 6. Версионирование и GitHub

- Основная ветка всегда должна быть запускаемой или содержать только документы
  до начала реализации.
- Каждая принятая фаза фиксируется отдельным тегом или явно обозначенной точкой
  отката.
- Изменения публикуются небольшими атомарными коммитами с причиной изменения.
- Код приложения и пакеты навыков версионируются независимо.
- Секреты, локальные базы, пользовательские данные и большие выгрузки не
  публикуются.
- Каждый релиз связывает версии приложения, набора навыков и тестового корпуса.

## 7. Definition of Done для MVP

MVP считается готовым только если одновременно:

- приложение устанавливается по документации на чистую среду;
- DeepSeek и MCP диагностируются отдельно;
- обязательные вопросы по документации и данным проходят согласованный порог;
- ответы основаны на фактических источниках и не выдают технические ошибки как
  бизнес-ответ;
- составные и follow-up вопросы сохраняют смысл диалога;
- пакет навыков переносится в другой экземпляр без изменения приложения;
- новый совместимый навык можно добавить без выпуска новой версии приложения;
- каждый вызов навыка, LLM и MCP виден в диагностике;
- полный автоматический тестовый прогон проходит;
- отсутствуют блокирующие дефекты и неподтвержденные данные в контрольной
  выборке;
- исходники, документация, версия и точка отката опубликованы в GitHub.
