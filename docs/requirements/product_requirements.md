## 1. Исходные условия

- Целевая конфигурация: `1С:Управление торговлей (базовая), редакция 11`,
  релиз `11.5.27.56`, совместимость `8.3.27`.
- Файловая выгрузка конфигурации:
  `/Users/repnikov/Documents/УТ_Демо/config`.
- Основной источник документации MVP: встроенная справка целевой выгрузки.
- Внешние материалы разрешены только как дополнительные источники с явной
  маркировкой их происхождения и возможной неактуальности.
- Фактические данные получаются из локальной демонстрационной базы УТ через
  известный MCP в режиме только для чтения.
- Основная языковая модель: DeepSeek через API; параметры задаются окружением.
- Первая среда разработки и приемки: macOS. После приемки выполняется проверка
  на Windows.
- Исходный код публикуется атомарными коммитами в `main` закрытого репозитория
  `Repich/ChatBot_1C_v.7`.
- Код, архитектура и навыки версии 5 не переносятся. Ее вопросы, дефекты и
  выводы используются только как источники требований и тестовых сценариев.

## 2. Цели MVP

1. Создать с нуля работоспособного ассистента для зафиксированной типовой УТ.
2. Давать проверяемые ответы на распространенные вопросы по данным базы и
   встроенной документации.
3. Поставлять продукт с заранее подготовленным и испытанным каталогом
   атомарных возможностей.
4. Объединять несколько возможностей для ответа на составные вопросы.
5. Переносить новые и обновленные навыки между совместимыми экземплярами без
   обновления приложения.
6. Проверить гипотезу, что готовый контролируемый каталог надежнее создания
   запросов к 1С во время каждого диалога.
7. Сделать каждый ответ воспроизводимым и диагностируемым.
8. Получить основу, которую после MVP можно расширять новыми бизнес-областями и
   пакетами навыков.

## 3. Пользователи

| Пользователь | Основная потребность |
| --- | --- |
| Руководитель | Сводные показатели, рейтинги, динамика и отклонения |
| Менеджер по продажам | Заказы, реализации, клиенты, товары и цены |
| Менеджер по закупкам | Заказы поставщикам, поступления, цены и ожидаемые поставки |
| Складской сотрудник | Остатки, доступность, резервы, движения и перемещения |
| Финансовый специалист | Задолженность, денежные средства, выручка и прибыль |
| Консультант 1С | Ответы по функциональности УТ и причинам пользовательских ошибок |
| Автор навыков | Создание и перенос проверенных JSON-навыков |
| Тестировщик и оператор | Воспроизведение сценариев и анализ полного хода выполнения |

## 4. Job stories

1. Когда руководителю нужен показатель, он хочет задать вопрос обычным языком
   и получить проверяемое значение без знания структуры 1С.
2. Когда пользователь не знает точное название объекта, он хочет использовать
   привычную бизнес-терминологию.
3. Когда вопрос неоднозначен, пользователь хочет получить понятное уточнение,
   а не случайно выбранную трактовку.
4. Когда вопрос относится к нескольким областям, пользователь хочет получить
   единый ответ без ручного объединения данных.
5. Когда пользователь продолжает диалог, он хочет ссылаться на предыдущий
   результат словами «там», «этот клиент» или «только по нему».
6. Когда данных нет, пользователь хочет отличать подтвержденное отсутствие
   данных от ошибки запроса или внешнего сервиса.
7. Когда возникает ошибка, консультант хочет получить идентификатор операции и
   достаточную диагностику для воспроизведения.
8. Когда подготовлен новый навык, оператор хочет перенести JSON-файл в другой
   совместимый экземпляр и сразу использовать его.
9. Когда документация противоречива, пользователь хочет видеть источник и
   предупреждение о возможной неактуальности.
10. Когда вопрос выходит за рамки УТ, пользователь хочет получить короткий
    понятный отказ без выдуманного ответа.

## 5. Функциональные требования

### 5.1. Диалог

- `FR-001`. Ассистент принимает русскоязычные вопросы через web-интерфейс.
- `FR-002`. История и контекст сохраняются в пределах пользовательской сессии.
- `FR-003`. Ассистент понимает перефразировки, распространенные опечатки и
  принятую бизнес-терминологию.
- `FR-004`. Ассистент различает вопросы по данным, документации, составные и
  нерелевантные вопросы.
- `FR-005`. Пользователю не требуется знать метаданные, язык запросов 1С, MCP
  или устройство приложения.
- `FR-006`. При существенной неоднозначности задается один конкретный
  уточняющий вопрос с понятными вариантами ответа, когда это уместно.
- `FR-007`. Ассистент не выбирает за пользователя бизнес-трактовку показателя,
  если доступны несколько существенно разных трактовок.
- `FR-008`. Продолжение диалога сохраняет явно заданные ранее объект, период,
  склад, товар и другие применимые условия.
- `FR-009`. Пользователь видит время отправки, время ответа и состояние
  длительной операции.
- `FR-010`. Базовые вопросы «кто ты» и «что ты умеешь» получают короткий ответ
  в границах фактических возможностей MVP.

### 5.2. Работа с данными

- `FR-011`. Фактические данные УТ получаются только через доступный MCP.
- `FR-012`. Все операции с данными в MVP выполняются только на чтение.
- `FR-013`. Для ответа выбираются подходящие готовые возможности каталога.
- `FR-014`. Составной вопрос может использовать несколько возможностей.
- `FR-015`. Значения из вопроса передаются как параметры и не становятся
  причиной создавать отдельный узкий навык.
- `FR-016`. Поддерживаются явные и относительные даты, интервалы и группировка
  по календарным периодам.
- `FR-017`. Поддерживаются списки, количество, суммы, средние значения,
  минимумы, максимумы и рейтинги.
- `FR-018`. Ответ различает число строк, число документов, количество единиц,
  число разных товаров и денежную сумму.
- `FR-019`. Для денежных значений указывается валюта либо прямо сообщается, что
  ее нельзя однозначно определить.
- `FR-020`. Для остатков указывается момент времени, для оборотов — период.
- `FR-021`. Пустой результат считается отсутствием данных только после
  подтвержденного успешного выполнения корректной выборки.
- `FR-022`. Промежуточные данные не выдаются за окончательный ответ.
- `FR-023`. Ассистент не придумывает значения при отсутствии данных,
  недоступности зависимости или ошибке выполнения.
- `FR-024`. Списки имеют понятный лимит, а пользователь может запросить
  продолжение, изменение лимита или полную выборку.
- `FR-025`. Итоговый ответ называет показатель, объект, период, единицу
  измерения и существенные условия отбора.

### 5.3. Работа с документацией

- `FR-026`. Ассистент объясняет назначение объектов и терминов УТ.
- `FR-027`. Ассистент описывает порядок выполнения типовых операций.
- `FR-028`. Ассистент ищет причины типовых ошибок и функциональных ограничений.
- `FR-029`. Документационный ответ содержит название использованного материала
  и место его нахождения в доступном корпусе.
- `FR-030`. Сведения из документации явно отделяются от фактических данных
  информационной базы.
- `FR-031`. При противоречии источников показываются предупреждение,
  происхождение и доступная дата каждого источника.
- `FR-032`. Встроенная справка целевого релиза имеет приоритет над внешними
  материалами.
- `FR-033`. Предположение модели не выдается за подтвержденную инструкцию.

### 5.4. Навыки как поставляемый продукт

- `FR-034`. MVP поставляется с готовым каталогом навыков для целевой УТ.
- `FR-035`. Один навык предоставляет одну устойчивую бизнес-возможность и
  принимает изменяемые параметры.
- `FR-036`. Навык не содержит конкретные товары, клиентов, склады,
  идентификаторы сессий, секреты или локальные пути.
- `FR-037`. Переносимое представление навыка содержит назначение, входные
  параметры, смысл результата, совместимость, версию, зависимости, примеры и
  контрольные проверки.
- `FR-038`. Навык экспортируется в отдельный JSON-файл.
- `FR-039`. Навык импортируется в другой совместимый экземпляр без обновления
  приложения.
- `FR-040`. Импортированный навык доступен для следующего пользовательского
  запроса без перезапуска приложения.
- `FR-041`. До применения проверяются формат, совместимость, обязательные
  сведения и конфликт версий.
- `FR-042`. Некорректный или несовместимый навык отклоняется с понятным
  перечнем причин.
- `FR-043`. Пользователь может просмотреть назначение, параметры, ожидаемый
  результат и примеры применения каждого навыка.
- `FR-044`. Неправильный навык можно удалить или заменить независимо от версии
  приложения.
- `FR-045`. Для каждого ответа сохраняется перечень фактически использованных
  возможностей.

### 5.5. Web-интерфейс и диагностика

- `FR-046`. Основная площадь интерфейса отведена чату, а строка ввода всегда
  доступна.
- `FR-047`. Сообщение отправляется кнопкой или клавишей Enter.
- `FR-048`. Сессии сохраняются и доступны после перезагрузки страницы.
- `FR-049`. Доступны просмотр каталога навыков, импорт и экспорт JSON.
- `FR-050`. Доступна диагностика состояния DeepSeek и MCP.
- `FR-051`. Для каждого запроса фиксируются вопрос, выбранные возможности,
  параметры, обращения к MCP, сырой ответ, итог и ошибки.
- `FR-052`. Ключи и секреты не отображаются в интерфейсе и диагностике.
- `FR-053`. Диагностика одного обращения выгружается отдельным файлом.
- `FR-054`. В интерфейсе отображаются версия продукта и состояние внешних
  зависимостей.
- `FR-055`. При ошибке DeepSeek, MCP или навыка пользователь получает понятное
  сообщение и идентификатор операции.

## 6. Нефункциональные требования

- `NFR-001`. Ответы и элементы пользовательского интерфейса выполняются на
  русском языке.
- `NFR-002`. Числа, даты, единицы и денежные значения имеют единообразное
  человекочитаемое представление.
- `NFR-003`. Один вопрос при неизменном состоянии данных должен иметь одинаковый
  фактический результат, даже если текст ответа незначительно различается.
- `NFR-004`. Ошибка отдельного запроса не приводит к потере сессии или остановке
  приложения.
- `NFR-005`. Журнал позволяет восстановить ход выполнения без догадок и без
  повторного запроса пользователя.
- `NFR-006`. Каталог навыков проверяется автоматически перед поставкой.
- `NFR-007`. Каждая продуктовая возможность имеет положительные, отрицательные
  и, где применимо, композиционные испытания.
- `NFR-008`. Ранняя разработка и приемка выполняются на macOS, затем тот же
  продукт и пакет навыков проверяются на Windows.
- `NFR-009`. Обновление каталога навыков не требует новой сборки приложения.
- `NFR-010`. Базовый вопрос должен завершаться не более чем за 30 секунд при
  нормальной работе зависимостей; 95% поддерживаемых вопросов — не более чем за
  90 секунд.
- `NFR-011`. При недоступности зависимости пользователь получает контролируемый
  ответ не позднее установленного таймаута.
- `NFR-012`. Исходники, требования, инструкции, история версий и тесты хранятся
  в GitHub и изменяются атомарными коммитами.
- `NFR-013`. Проект не содержит перенесенного кода, архитектуры или формата
  навыков версии 5.
- `NFR-014`. Базовая эксплуатационная гигиена прототипа включает хранение
  ключей только в окружении и исключение секретов из Git и журналов.

## 7. Границы MVP

### Входит в MVP

- целевая УТ 11.5.27.56 без отраслевых расширений;
- локальная демонстрационная база с фиксируемым маркером состояния;
- DeepSeek как единственная основная LLM;
- чтение данных через MCP;
- ответы по встроенной справке целевого релиза;
- дополнительные внешние материалы с явной маркировкой;
- готовый каталог переносимых JSON-навыков;
- сочетание готовых возможностей и многошаговый диалог;
- уточнения, подтвержденное отсутствие данных и контролируемые ошибки;
- web-интерфейс, сохраненные сессии и диагностические журналы;
- импорт, экспорт, замена и удаление навыков;
- автоматические и ручные испытания на контрольной базе;
- поставка для macOS и последующая проверка Windows.

### Не входит в MVP

- поддержка произвольной неизвестной конфигурации 1С;
- автономное создание новых навыков во время пользовательского диалога;
- использование кода или устройства приложения версии 5;
- создание, изменение и проведение данных в 1С;
- поддержка отраслевых расширений и доработок заказчиков;
- голосовой интерфейс, мессенджеры и мобильное приложение;
- обучение собственной языковой модели;
- полноценное прогнозирование и оптимизация закупок;
- промышленная высокая доступность и высокая нагрузка;
- отдельная корпоративная модель угроз на этапе проверки гипотезы.

## 8. Правило эталонных данных

- Эталоны для вопросов по данным формируются автоматически на локальной
  демонстрационной базе независимыми контрольными выборками.
- Вместе с эталонами фиксируется маркер состояния базы, позволяющий определить,
  что данные после построения эталона изменились.
- Числовой или списочный эталон нельзя переносить на другое состояние базы без
  повторного формирования.
- Отсутствие доступного живого MCP не препятствует дальнейшему анализу и
  архитектурной проработке, но препятствует формированию фактических эталонов и
  выполнению живых интеграционных испытаний.
